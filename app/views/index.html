1. Design and test

From testing your service with users, you may see the need for a new pattern or to change a existing one.

You may have observed one or more of the following:

  • a unmet user need
  • same problem is trying to be solved on at least 2 different services
  • A/B testing a existing pattern and your design

2. Blog about research and testing

Every pattern is published with user research and usability testing supporting it.

Blogging is a good way to document and share this work, as shows what was involved and when it was done, while protecting users privacy.

When you blog about the work supporting a pattern include:

  • user need of the service you were testing
  • how many users you tested with
  • how many rounds of research you did
  • what further testing is needed

3. Publish work for others to see

Part of something becoming a pattern is it being discussed and reviewed.

For this to happen the work, be it code, words or a prototypes, needs to publish so others can see and interact with it.

Publish:

  • code in a public repository on GitHub
  • prototypes on Heroku, without a password
  • research and testing on a government blog, if possible

4. Discuss with community

Patterns come about through consensus.

Create or edit a page in the patterns wiki, so the community can discuss it.

Let the community know about the pattern, for example via the mailing list or meet ups.

Use the guidance for how to maintain the patterns wiki.

5. Submit pattern for review

To ensure qualify, patterns are reviewed before they go onto service manual.

To submit a pattern include:

  • short description of the user need the pattern meets
  • link to blogpost about research and testing
  • link to code
  • link to prototype
  • link to discussion about the pattern

Email these to service manual team

You'll be notified on when your pattern will be reviewed.

6. How patterns are reviewed

Every month a group meet to review patterns to be published.

This group includes:

  • Two designers and/or developers in departments
  • GDS designer
  • GDS content designer
  • GDS user researcher
  • GDS accessibility
  • GDS frontend developer

Group decides:

  • new patterns to publish
  • if a new pattern needs more work
  • to change existing patterns

In order for decisions are consistent, there are standards to inform them.

A standard pattern must have:

  • been through usability research
  • accessible to WCAG AA
  • been in use in a service
  • has no major unresolved discussions

7. Publish on service manual

All contributors to a pattern are published on service manual.

After the decision to publish a pattern, the service manual team writes content on when and how to use it. This may involve working with it's contributors.

Patterns may need to be iterated from the ones submitted, to make them consistent and accessible.
